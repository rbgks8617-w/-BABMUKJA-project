# 보안 설계

대학교 밥먹자는 학생 정보, 주문 정보, 결제 정보를 다루는 서비스입니다. 상용화 단계에서는 보안을 기능 개발과 같은 우선순위로 다룹니다.

## 핵심 원칙

1. 카드 정보를 직접 저장하지 않습니다.
2. 앱에서 보낸 가격을 신뢰하지 않습니다.
3. 결제 완료 여부는 서버가 PG API로 검증합니다.
4. 학생은 자기 주문만 볼 수 있습니다.
5. 식당 관리자는 자기 식당 주문만 볼 수 있습니다.
6. 관리자 작업은 감사 로그에 남깁니다.
7. API 키와 비밀키는 GitHub에 올리지 않습니다.

## 결제 보안

절대 저장하지 않는 정보:

- 카드번호 전체
- CVC
- 카드 비밀번호
- 마그네틱/칩 데이터

저장 가능한 정보:

- PG 결제키
- 주문번호
- 승인 금액
- 결제 상태
- 카드사 코드
- 마스킹된 카드 정보
- 빌링키 또는 토큰

결제 흐름:

1. 앱이 서버에 주문 생성을 요청합니다.
2. 서버가 DB 기준 메뉴 가격으로 주문 금액을 다시 계산합니다.
3. 서버가 `PENDING_PAYMENT` 주문을 생성합니다.
4. 앱이 PG 결제창을 호출합니다.
5. 결제 성공 후 앱이 결제키를 서버에 전달합니다.
6. 서버가 PG API로 주문번호, 결제키, 금액을 검증합니다.
7. 검증 성공 시 주문 상태를 `PAID`로 변경합니다.
8. 검증 실패 시 주문을 실패 상태로 변경하고 사용자에게 안내합니다.

## 인증 보안

학생 인증:

- 학교 이메일 인증
- OAuth 로그인
- refresh token 보호

관리자 인증:

- 강한 비밀번호 정책
- 2단계 인증 고려
- 관리자 권한 분리

권한 모델:

- `STUDENT`
- `RESTAURANT_ADMIN`
- `SERVICE_ADMIN`

## API 보안

필수 적용:

- HTTPS
- JWT 또는 세션 기반 인증
- rate limit
- 요청 body validation
- 서버 기준 가격 계산
- 권한 체크
- CORS 제한
- 관리자 API 별도 보호

예시:

- 학생이 `/orders/:id`를 조회할 때 `order.userId === currentUser.id`인지 확인합니다.
- 식당 관리자가 주문 상태를 바꿀 때 `order.restaurantId`가 관리자 소속 식당인지 확인합니다.
- 결제 승인 요청 금액은 클라이언트 값이 아니라 서버 계산값을 사용합니다.

## DB 보안

필수 원칙:

- 운영 DB 직접 접근 제한
- 최소 권한 계정 사용
- 정기 백업
- 개인정보 컬럼 접근 제한
- 감사 로그 저장

감사 로그 대상:

- 메뉴 가격 변경
- 품절 처리
- 주문 상태 변경
- 환불 처리
- 관리자 로그인 실패
- 관리자 권한 변경

## 모니터링 보안

탐지해야 할 이벤트:

- 로그인 실패 폭증
- 같은 IP에서 과도한 요청
- 결제 승인 실패율 증가
- 주문 생성 실패율 증가
- 관리자 API 반복 호출
- 비정상적으로 높은 환불 요청

자동 대응 후보:

- IP 임시 차단
- rate limit 강화
- 관리자 세션 강제 만료
- 알림 발송

수동 확인 필요:

- 결제 장애
- 환불 분쟁
- 개인정보 노출 의심
- 관리자 계정 탈취 의심

## 배포 보안

필수:

- `.env` 파일 GitHub 업로드 금지
- 운영/개발 키 분리
- PG secret key 서버에만 저장
- DB 비밀번호 환경변수 관리
- HTTPS 강제
- 백업과 복구 절차 문서화
