# 상용화 로드맵

이 문서는 대학교 밥먹자 앱을 프로토타입에서 실제 상용 서비스로 발전시키기 위한 실행 계획입니다.

## 1단계: 현재 앱 틀 완성

목표는 친구, 학생, 학교 관계자에게 보여줄 수 있는 수준의 프로토타입을 안정화하는 것입니다.

- 전체 화면 UI 정리
- 장바구니 UX 정리
- 메뉴 추천 UX 정리
- 주문 완료 화면 정리
- 앱 폰트, 색상, 버튼 스타일 통일
- README와 문서 정리
- APK 테스트 빌드 유지
- 웹 공유 링크 유지

완료 기준:

- 앱을 처음 보는 사람이 식당 선택, 메뉴 선택, 장바구니, 결제 더미, 주문 완료까지 막힘 없이 진행할 수 있어야 합니다.

## 2단계: TypeScript 전환

상용화 단계에서는 주문, 결제, 인증처럼 실수가 치명적인 코드가 많아집니다. JavaScript에서 TypeScript로 전환해 데이터 구조를 명확히 합니다.

- `App.js`를 `App.tsx`로 전환
- 화면 컴포넌트 타입 정의
- 메뉴, 식당, 장바구니, 주문 타입 정의
- navigation param 타입 정의
- 서비스 함수 응답 타입 정의

완료 기준:

- 식당, 메뉴, 장바구니, 주문 데이터가 타입으로 보호되어야 합니다.

## 3단계: 백엔드 설계

초기 추천 구조:

- NestJS
- PostgreSQL
- Prisma
- Redis

핵심 API:

- 학생 회원가입/로그인
- 학교 인증
- 식당 목록 조회
- 메뉴 목록 조회
- 장바구니 검증
- 주문 생성
- 결제 준비
- 결제 승인 검증
- 주문 상태 변경
- 환불 요청
- 관리자 메뉴 관리

완료 기준:

- 앱 더미 데이터를 제거하고 서버 API에서 식당/메뉴/주문 데이터를 받을 수 있어야 합니다.

## 4단계: DB 설계

핵심 테이블:

- users
- universities
- restaurants
- menus
- menu_options
- carts 또는 cart_items
- orders
- order_items
- payments
- refunds
- restaurant_admins
- audit_logs

중요 원칙:

- 앱에서 보낸 가격을 그대로 믿지 않습니다.
- 주문 금액은 서버 DB 기준으로 다시 계산합니다.
- 결제 완료 전 주문은 `PENDING_PAYMENT` 상태로 둡니다.
- 결제 검증 완료 후에만 `PAID`로 변경합니다.

## 5단계: 인증

MVP 인증 후보:

- Firebase Auth
- Supabase Auth

대학 전용 인증:

- 학교 이메일 인증
- 학교 선택
- 학생 프로필 생성

권한:

- 학생
- 식당 관리자
- 서비스 관리자

## 6단계: 결제

추천 PG:

- 토스페이먼츠
- 포트원

원칙:

- 카드번호, CVC, 카드 비밀번호를 앱이나 서버에 저장하지 않습니다.
- 결제창/SDK는 PG가 제공합니다.
- 결제 승인 검증은 반드시 서버에서 합니다.
- 서버는 주문번호, 결제키, 금액을 PG API로 다시 검증합니다.

## 7단계: 관리자 페이지

식당 관리자가 필요한 기능:

- 메뉴 추가/수정/삭제
- 품절 처리
- 운영 상태 변경
- 주문 접수
- 조리중 처리
- 픽업 가능 처리
- 주문 완료 처리

관리자 페이지 후보:

- Next.js
- React Admin

## 8단계: 배포

초기 MVP 배포 후보:

- 앱: EAS Build
- 백엔드: Render, Railway, Fly.io, AWS ECS 중 선택
- DB: Supabase Postgres 또는 AWS RDS
- 정적 관리자 페이지: Vercel

상용화 확장 후보:

- AWS ECS/Fargate
- AWS RDS PostgreSQL
- AWS ElastiCache Redis
- CloudFront
- WAF

## 9단계: 모니터링과 자동 대응

필수 모니터링 지표:

- API 응답 시간
- 에러율
- CPU/메모리
- DB 연결 수
- 주문 생성 실패율
- 결제 승인 실패율
- 비정상 로그인 시도
- 과도한 요청 IP

도구:

- Sentry
- Grafana/Prometheus
- CloudWatch 또는 배포 플랫폼 기본 모니터링
- Discord/Slack 알림

자동 대응:

- 서버 재시작
- 인스턴스 확장
- rate limit 강화
- 의심 IP 임시 차단
- 장애 알림 발송

사람 확인이 필요한 대응:

- 결제 오류
- 환불 분쟁
- 개인정보 사고 의심
- 관리자 계정 탈취 의심

## 10단계: 베타 테스트

처음부터 전국 대학으로 확장하지 않습니다. 한 학교, 일부 식당, 제한된 학생 그룹으로 시작합니다.

베타 테스트 체크리스트:

- 주문 흐름이 이해되는가
- 식당이 주문을 놓치지 않는가
- 메뉴 품절/운영 상태가 잘 반영되는가
- 결제 실패 시 복구가 되는가
- 환불/취소 문의가 처리 가능한가
